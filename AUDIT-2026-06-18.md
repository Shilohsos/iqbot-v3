## PART 1: CRITICAL FINDINGS

### C1. Missing DB Migrations — Fresh Deploy WILL Break
**Source:** auto-trading.ts audit, line references to db.ts

The `auto_trading_sessions` table DDL (db.ts lines 181-199) does NOT include `mg_active`, `mg_next_amount`, or `status_msg_id` columns. The lazy migration only adds `evaluations` and `mode`. These three columns exist in the current DB from a prior version but have NO ALTER TABLE migration. On a fresh database:

- `setAutoSessionMgState()` → "no such column: mg_active"
- `renderStatus()` → "no such column: status_msg_id"  
- `INSERT INTO auto_trading_sessions` → fails

**Fix:** Add ALTER TABLE statements in db.ts migration block.

### C2. No Circuit Breaker — 102 ERROR Trades in 2 Minutes
**Source:** DB audit, user 1341582495

On June 17, user 1341582495 generated 102 ERROR trades between 14:45-15:31 UTC. All with `trade_id=0` (trade never placed). The auto-trading loop kept retrying without detecting the persistent failure. Claude's fix today added `MAX_CONSECUTIVE_ERRORS=3` — this gap is now CLOSED for future runs. But the 102 error cascade already happened and the duplicate trade_id entries remain in the DB.

**Status:** FIXED in code (b0fd799), DB cleanup still needed.

### C3. Orphaned Martingale State — Risk of Unintended Trades
**Source:** DB audit, auto_trading_sessions table

Two STOPPED sessions have `mg_active=1` with pending recovery amounts:

| Session | User | Status | mg_next_amount | Risk |
|---------|------|--------|---------------|------|
| 5 | 8986669286 | stopped | 40,000 NGN | If resumed, fires 40k trade |
| 40 | 6360621016 | stopped | 400 | If resumed, fires 400 trade |

If any code path checks `mg_active` without verifying `status='running'`, these could trigger unintended martingale recovery trades. Claude's fix clears orphaned mg_active on `start()` — but only for sessions being started. These stopped sessions won't be cleared until they're restarted.

**Fix:** Run cleanup on these two rows now. Add startup migration to clear mg_active on all non-running sessions.

### C4. setAutoSessionMgState Can Bypass Error Handler
**Source:** auto-trading.ts audit, line 475

In the trade error catch block, `setAutoSessionMgState(false)` is called BEFORE the reconnect logic, consecutiveErrors increment, and user notification. If this DB call throws (schema gap, deadlock), the ENTIRE error handling block (reconnect, retry, notify) is skipped. The error propagates to the outer catch which silently pauses the session with no user notification.

**Fix:** Move setAutoSessionMgState to AFTER the reconnect/user-notify logic, or wrap in its own try/catch.

## PART 2: HIGH SEVERITY

### H1. Position Check Silently Masks Non-Timeout Errors
**Source:** auto-trading.ts audit, lines 386-392

The positions check regex only catches `^TIMEOUT:`. Auth failures, WS crashes, and SDK errors are silently treated as "no open position." The loop continues for a full iteration before the analysis stage catches the error 15s later. Wastes time, not a crash — but masks real issues.

### H2. connect() Has No Timeout in start()
**Source:** auto-trading.ts audit, line 301

`createSdk()` inside `connect()` has no timeout wrapper. A hung WebSocket handshake blocks `start()` indefinitely. The `.catch()` on the loop launch promise won't fire because `start()` itself hasn't returned.

### H3. Pause→Resume Race Can Orphan Session
**Source:** auto-trading.ts audit, concurrency analysis

If `resume()` is called before the paused loop's while-condition re-checks, `start()` sees existing runner and returns early. The old loop exits, unregisters, and no new runner is created. Session stays 'running' with no runner.

### H4. 172 Users Have Trades but $0 funded_balance
**Source:** DB audit, users table

172 users have executed real trades but show `funded_balance_usd=0`. The stale-balance bug from June 16 (commit 41add2f attempted fix) is still affecting these users. Their actual IQ Option balances are unknown without live SDK checks.

### H5. 78 Users Have mg_enabled=1 but No Session
**Source:** DB audit, users table

78 users have `mg_enabled=1` in their profile but no entry in `auto_trading_sessions`. If any code path checks this flag to trigger session creation, these are ghost entries. Unclear if this is by design or stale data.

### H6. 102 Duplicate Trade Entries
**Source:** DB audit, trades table

User 1341582495 has every trade recorded TWICE in the database — 102 pairs of duplicate `trade_id` entries. The system doesn't deduplicate on insert. This inflates trade count and PnL for this user.

## PART 3: MEDIUM SEVERITY

### M1. No Indexes on Critical Tables
auto_trading_sessions: no indexes (full table scans on status/mg_active queries)  
signal_tracking: no indexes at all (2,789 rows, growing)  
users: minimal indexing (no index on ssid_valid, access_level, mg_enabled)

### M2. Stale Session Accumulation
24 interactive session entries older than 1 day (abandoned flows). No TTL/cleanup mechanism. 11 stale auto_trading_sessions (stopped/paused >1 day).

### M3. flushDemoMinutes Silently Swallows DB Errors
Empty `catch {}` on line 66 of auto-trading.ts. Demo minutes could be lost without detection.

### M4. Proxy Cron Rotation vs In-Process Conflict
Cron does `pm2 restart` (drops all sessions) on rotation. In-process rotation avoids restart. If cron detects stale .env after in-process rotation, it triggers a disruptive restart. Cron should check before restarting.

### M5. WS Proxy is Patch, Not SDK-Native
The `wsProxyResolver` mechanism injects proxy support via a global function set during bot startup. If the Quadcode SDK is updated, this patch may break. Consider upstreaming or maintaining a file-level diff.
